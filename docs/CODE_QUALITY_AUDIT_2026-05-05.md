## 7. Lint config gaps

Current config (`eslint.config.mjs`) applies the same rules to test and non-test files. Findings:

- `@typescript-eslint/no-non-null-assertion: warn` is correctly set, but tests legitimately use `!` on fixtures and shouldn't trigger it. **435 of the 507 warnings are this rule firing inside tests.** Disabling it for `**/*.test.ts` would reduce noise to a meaningful number (74) without lowering the production bar.
- `complexity` rule is **not configured at all.** Adding it at threshold 30 (warn) would surface the complexity offenders and prevent regression. Adding it at threshold 50 (error) would gate the worst offenders.
- `max-lines` rule not configured. Adding `["warn", 800]` would put a tripwire on future god files.
- `@typescript-eslint/no-explicit-any: off` is set, with rationale comment. That's fine — but at the file level, the high-`any` files (dashboard, spark) deserve a path-scoped exemption rather than a project-wide off.

2. Drop `cli.ts:main` complexity below 30 by extracting subcommand handlers.

4. Lift coverage on `solve.js` and `mcp.js` above 60%.

- Splitting `dashboard/command-api.ts` (it's an HTTP handler file; size is inherent).
- Splitting `tools/graphics.ts` (specialized; size justifiable until proven otherwise).
- Cosmetic warning cleanup (the 74 production warnings are mostly `no-unused-vars` on dev imports — safe, not load-bearing).

## 11. Anti-theater commitments for the fix work

For each fix above, the bar is:

- Baseline: the metric value before the change (already in this doc).
- After: the metric value re-measured by the same command.
- Both numbers in the commit body. No claim of improvement without paired measurements.
- Test suite passes 2,205/2,205 after every commit. No regressions.
- App sync via post-commit hook completes; verified by mtime + version on disk.

If a fix doesn't improve the metric or breaks tests, it gets reverted, not "spun as partial success."
