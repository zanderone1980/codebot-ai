# RFC 001 — Better file localization (Tier 3.2)

**Status**: design, not built
**Author**: Claude (audit session 2026-04-17)
**Estimated effort**: 1–2 weeks
**Expected impact**: +3–10pp on SWE-bench Verified (mid-tier estimate)

## Problem

In the 50-task SWE-bench run, patch sizes for resolved (avg 2003 B) and
unresolved (avg 2351 B) were nearly identical. Failures aren't from
patches being too short or too long — they're from patches being in the
**wrong place**: wrong file, right file but wrong function, missing the
codepath that matters.

Example: `django-11400` produced a patch touching `admin/filters.py`
when the actual fix needed coordination across `model_fields.py` and
the `RelatedFieldListFilter` ordering machinery. CodeBot's localization
relied entirely on the LLM reading the problem statement, then `grep`/
`read_file` blindly until it found something plausible.

This is the dominant failure mode for the 16 wrong-patch unresolved tasks.

## Current behavior

CodeBot's tool registry has `read_file`, `grep`, `glob`, `find`, but no
structural understanding of the repo. The agent navigates by:
1. Reading the problem statement
2. Grepping for keywords from the problem
3. Reading whatever files come back
4. Sometimes following imports manually

This works when problem statements name the bad file (e.g., "the bug is
in `astropy/modeling/separable.py`"). It fails when the problem describes
behavior and the model has to infer the source location.

## Proposal

Three concrete additions, in priority order:

### A. Symbol index per session (~3 days)

Build a flat symbol-to-location index when CodeBot opens a project:
```
{
  "django.db.models.fields.related.RelatedFieldListFilter": "django/db/models/fields/related.py:1247",
  "RelatedFieldListFilter.field_choices": "django/db/models/fields/related.py:1289",
  ...
}
```

New tool: `find_symbol(name: str) -> [{file, line, kind}]`. CodeBot can
call `find_symbol("RelatedFieldListFilter")` instead of grepping for the
string in the entire repo and getting 50+ hits.

Implementation: ripgrep-based with simple regex per language
(`def name(`, `class name`, `function name`, `const name =`, etc.).
Cached per-project at `~/.codebot/symbols/<project-hash>.json`.

This is the highest-ROI part of the RFC. Top open-source agents
(Aider, SWE-agent) have variants of this.

### B. Repo structure summary in system prompt (~1 day)

Currently the agent sees only the user's problem statement. Add a
machine-generated overview at session start:

```
Repository: django/django
Top-level packages:
  django/        Django framework source
    contrib/     Bundled apps (admin, auth, etc.)
    db/          ORM (models, fields, query, sql)
    forms/       Form rendering and validation
    template/    Template engine
    test/        Test framework
  tests/         Project test suite
  docs/          Sphinx docs
```

Generated by walking the top 2 levels of directories and pulling each
package's `__init__.py` docstring or first non-blank comment. ~50 lines
of code, runs in <1s on cold cache.

Token cost: ~500-1000 tokens per session, paid once. Negligible vs
typical 100K+ session token spend.

### C. Call-graph awareness (~1 week, future)

When CodeBot edits function `foo`, automatically surface its callers
("functions that call `foo` in this repo"). Helps catch "I fixed the
function but missed the call site that needed updating" errors.

Implementation: tree-sitter-based call graph extracted on-demand. More
complex than A; defer until A and B are in place and we have data on
where they fall short.

## Validation plan

Before merging:
1. Re-run the 50-task SWE-bench slice with A+B
2. Compare resolved count vs the 17/50 baseline
3. Look at the 16 previously-wrong-patch tasks specifically — did
   localization improve? (Manual diff inspection on 5-10 of them)

A 50-task A/B run costs ~$15-20 and ~3 hours wall. Worth it before
declaring success.

## Non-goals

- Embeddings-based semantic search. Heavy infra (vector DB, embedding
  model selection, indexing pipeline) for marginal lift over A+B.
- Full LSP integration. Massive scope; defer indefinitely.
- AST-based refactoring. Different problem (we're solving "where to
  edit", not "how to edit").

## Risks

- **A** can over-fire on common names (`get`, `update`, `save`). Mitigation:
  return up to N matches with file paths so the agent picks.
- **B** can leak stale info if the repo has dead packages. Mitigation:
  filter to packages that actually have non-test code in them.

## Open questions

1. Per-language symbol regex coverage. Python and JS/TS are easy; Rust /
   Go / Ruby need work.
2. Cache invalidation. When does a symbol index go stale? Heuristic: rebuild
   on git HEAD change.
3. Should `find_symbol` be a separate tool or absorbed into `grep` with a
   `--symbol` flag? Probably separate — different mental model.
